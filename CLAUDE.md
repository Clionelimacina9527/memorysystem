## Commands

```bash
# Build index.js from source files (required before deploy/dev)
npm run build        # or: node build.js

# Local development
npm run dev          # builds then runs wrangler dev

# Deploy to Cloudflare Workers
npm run deploy       # builds then runs wrangler deploy

# Tail production logs
npx wrangler tail
```

All sensitive config (secrets) is managed via Cloudflare Secrets, not `wrangler.toml`:
```bash
npx wrangler secret put JWT_SECRET
npx wrangler secret put HASH_SECRET
npx wrangler secret put INVITE_CODE
npx wrangler secret put ADMIN_EMAIL
npx wrangler secret put FEISHU_HOOK
```

## Source structure

```
src/
  worker.js       Backend Worker logic (API routes, DB, auth)
  frontend.html   Full frontend SPA (HTML/CSS/JS, properly formatted)
build.js          Node.js build script — inlines frontend.html into worker.js → index.js
package.json      npm scripts (build / dev / deploy)
index.js          Build artifact — the actual deployed file (git-tracked)
wrangler.toml     Cloudflare config (points to index.js)
```

**Never edit `index.js` directly.** Edit `src/worker.js` or `src/frontend.html`, then run `npm run build`.

## Architecture

This is a **single-file Cloudflare Worker** — the deployed `index.js` is generated by `build.js` which prepends the HTML as `const HTML_CONTENT = <json-string>` before the worker source.

- **Runtime**: Cloudflare Workers (edge, no Node.js APIs — uses Web Crypto, `fetch`, etc.)
- **Database**: Cloudflare D1 (SQLite), bound as `env.DB`
- **Frontend**: Full SPA served at `GET /` — `HTML_CONTENT` string returned as `text/html`

### Request handling

`src/worker.js` export default routes by matching `path` and `method` directly — no router framework. All API routes are under `/api/`.

### Auth

JWT-based: HMAC-SHA256, 7-day expiry. `signJWT`/`verifyJWT` use Web Crypto directly. Passwords use PBKDF2 (100k iterations, SHA-256). Legacy SHA-256 passwords are migrated on next login. Login rate-limited to 5 attempts per 15 minutes per email via `login_attempts` table.

### Data model

| Table | Purpose |
|-------|---------|
| `users` | Registered users — has `notif_last_seen INTEGER` column for notification read state |
| `worklogs` | Daily work records (done/plan/problem/thinking/important fields) |
| `projects` | Tags (UI calls them "标签", DB table is `projects`) |
| `project_entries` | Join table linking worklogs to a tag — one tag per worklog in practice |
| `comments` | Comments on worklogs |
| `login_attempts` | Rate limiting |

`setWorklogTag` replaces a worklog's tag: deletes existing `project_entries` rows then inserts a new one, creating the tag in `projects` if needed.

### File attachments (Cloudflare Workers KV)

KV namespace `memory-uploadfiles` is bound as `env.KV`. Files stored with a UUID key; metadata (filename, MIME type) stored in KV metadata and in the `attachments` D1 table.

- `POST /api/worklogs/{id}/attachments` — multipart upload, max 5 MB, allowed: `image/*` + `application/pdf`
- `GET /api/files/{key}` — serves file from KV with correct `Content-Type`; **no auth** (UUID is the capability — 122 bits of entropy)
- `DELETE /api/attachments/{id}` — deletes from KV and D1 (owner or admin only)
- Deleting a worklog also purges all its KV files first

| Table | Purpose |
|-------|---------|
| `attachments` | Metadata per file: `worklog_id`, `file_key` (KV UUID), `file_name`, `file_size`, `file_type`, `author_id` |

### Pagination (server-side)

`GET /api/worklogs` accepts `page`, `pageSize` (max 100, default 50), `author`, `dateFrom`, `dateTo`, `keyword`. Returns `{ results, total, page, pageSize, authors }`. Filtering and keyword search are handled in SQL with dynamic WHERE clauses. Frontend sends filter state as query params; a 350ms debounce applies to keyword input.

### Notifications (DB-persisted)

`users.notif_last_seen` (unix timestamp) tracks when the user last read notifications. `GET /api/my-comment-notifications` returns `{ items, unread_count }` where `is_new = created_at > notif_last_seen`. `POST /api/notifications/mark-read` sets `notif_last_seen = now`. Read state persists across devices/browsers. Feishu webhook is still fired fire-and-forget on comment creation.

### Frontend

Stores `memory_token` and `memory_user` in `localStorage`. All API calls use `Authorization: Bearer <token>`. Table rows are editable in-place. Pagination controls appear below the table when `totalPages > 1`. The `pictures/` directory is not referenced by the worker.
